# 03 Performance And Scaling Plan

## Performance Goal

The platform must feel fast for normal users and remain stable when many users open the system at once. The most important performance principle is to separate lightweight web requests from heavy raster processing.

A production-grade target is:

- Frontend first paint under 2 seconds on normal broadband.
- API health/status responses under 200 ms.
- Metadata/status reads under 500 ms.
- Preview image responses under 1 second after cache/CDN warmup.
- Long processing jobs handled asynchronously with progress updates.
- No user request should block for multiple minutes while processing GeoTIFFs.

## Current Performance Profile

### Fast Operations

These should be fast because they are mostly database reads or small JSON responses:

- `/health`
- `/api/v1/status`
- session list/read
- registered dataset list/read
- workflow status
- mission-control profile after outputs exist
- analytics overview
- export/package/report status reads

### Medium Operations

These may take seconds depending on dataset size:

- dataset inspection
- metadata extraction
- preview generation
- geospatial context generation
- location context generation
- temporal context packaging

### Heavy Operations

These are the main scaling risks:

- cloud probability map generation
- classification map generation
- cloud/shadow segmentation
- reconstruction
- reconstruction optimization
- evaluation metrics
- confidence estimation
- heatmap/overlay generation
- PDF report generation
- ZIP package generation

These should become queued background jobs in production.

## Primary Bottlenecks

### 1. Large Raster I/O

GeoTIFF files are large and disk-heavy. Reading full bands repeatedly will slow down processing and can overload memory.

Solutions:

- Use tiled/COG-style GeoTIFFs where possible.
- Read only required windows when doing partial previews.
- Use overviews/pyramids for visualization.
- Cache generated PNG previews.
- Avoid repeatedly reading the same bands in multiple services.
- Use a shared artifact registry so later stages reuse earlier outputs.

### 2. CPU-Bound Image Processing

OpenCV, NumPy, rasterio, and reconstruction operations can be CPU-heavy.

Solutions:

- Move heavy jobs to worker processes.
- Limit worker concurrency per machine based on CPU and memory.
- Use process-level parallelism instead of blocking FastAPI workers.
- Use job queues with retries and status tracking.
- Use separate worker pools for preview, cloud, reconstruction, export.

### 3. SQLite Write Locking

SQLite is fine locally but not for 500 simultaneous users.

Problems:

- One writer at a time.
- Hard to horizontally scale.
- Database file locking can conflict in containers.

Solutions:

- Migrate to PostgreSQL for production.
- Add PostGIS for spatial queries.
- Use indexes on `dataset_id`, `session_id`, status fields, and timestamps.
- Add Alembic migrations.
- Use connection pooling.

### 4. Local Filesystem Coupling

Current code stores many paths to local files. This is okay in local Docker, but not enough for distributed production.

Problems:

- If backend instances scale horizontally, one instance may not see files generated by another.
- Containers may lose local files during redeploy.
- Vercel cannot serve large backend-generated artifacts from backend local paths.

Solutions:

- Use S3/GCS/Azure Blob/R2 object storage.
- Store object keys in DB.
- Generate signed read URLs for private assets.
- Put public previews behind CDN.
- Use a shared temporary scratch disk only during job execution.

### 5. Frontend Image Loading

Large previews and overlays can slow UI pages.

Solutions:

- Always render thumbnail first.
- Lazy-load full previews.
- Use compressed PNG/JPEG/WebP previews for browser display.
- Keep GeoTIFF download separate from preview display.
- Use CDN headers for immutable generated assets.
- Avoid loading all subpage data at once.

## 500 Concurrent User Plan

500 users do not mean 500 simultaneous heavy reconstructions. The system should handle 500 users browsing and maybe a smaller number running heavy jobs.

Recommended capacity model:

```text
500 concurrent users
  -> 400 browsing/viewing cached outputs
  -> 80 reading API status/metadata
  -> 20 actively running heavy jobs
```

### Frontend

Use Vercel. It can comfortably serve the Next.js UI and cached static assets.

Frontend rules:

- Keep heavy files outside the frontend bundle.
- Fetch images from CDN/object storage.
- Poll workflow status at a controlled interval, for example every 3-5 seconds during active jobs.
- Stop polling after completion.
- Avoid polling all modules at once.

### API Backend

Recommended starting production configuration:

- 2-4 API replicas.
- Each API replica: 1-2 vCPU, 1-2 GB RAM minimum.
- Uvicorn workers: tune based on hosting platform. For CPU-bound API code, prefer fewer workers and move heavy work out.
- API handles validation, job creation, status reads, and signed URL generation.

### Worker Backend

Recommended starting configuration:

- Separate worker service from API.
- 2-4 worker replicas for hackathon-scale production.
- Each worker: 2-4 vCPU, 4-8 GB RAM if processing large rasters.
- Queue controls concurrency to prevent memory exhaustion.

Example pools:

```text
preview-worker:        concurrency 2-4
cloud-worker:          concurrency 1-2 per machine
reconstruction-worker: concurrency 1 per machine
export-worker:         concurrency 1-2
```

### Database

Use managed PostgreSQL.

Recommended minimum:

- 2 vCPU equivalent
- 4 GB RAM
- automated backups
- connection pooling enabled

Critical indexes:

- `analysis_sessions(session_id)`
- `datasets(dataset_id)`
- `datasets(analysis_session_id)`
- `dataset_metadata(dataset_id)`
- `dataset_previews(dataset_id)`
- all cloud tables on `dataset_id`
- reconstruction and fusion tables on `session_id`
- confidence tables on reconstruction/confidence/reliability ids
- exports on `session_id` and `export_id`

### Storage/CDN

Use object storage plus CDN.

- Raw GeoTIFFs: private bucket.
- Intermediate rasters: private bucket.
- Preview PNG/JPEG: public CDN or signed URL with caching.
- Reports/packages: signed download URLs.

Recommended cache headers:

- Generated immutable previews: long cache.
- Workflow JSON/status: no-store or very short cache.
- Static frontend assets: Vercel default immutable cache.

## API Responsiveness Plan

### Current Pattern

Many backend endpoints trigger work directly and return after completion. That is simple but not ideal for production.

### Production Pattern

Change long-running POST endpoints to this pattern:

```text
POST /run
  -> validate prerequisites
  -> create job row
  -> enqueue job
  -> return {job_id, status: QUEUED}

GET /workflow/{session_id}
  -> return current state and progress

GET /job/{job_id}
  -> return exact job status
```

This makes the API respond quickly even for slow raster operations.

## Progress Updates

Current frontend progress bars can be driven by sequential API calls. Production should support:

- Polling workflow endpoint every few seconds.
- Optional Server-Sent Events for live logs.
- Optional WebSocket channel for real-time progress.

For hackathon, polling is enough and easier to stabilize.

## Caching Strategy

### Backend Cache

Cache derived outputs by dataset/session:

- metadata extraction result
- preview image
- geospatial context
- location context
- cloud masks
- reconstruction previews
- confidence overlays
- reports/packages

Do not recompute if output exists and input checksum is unchanged.

### Input Checksums

Add a dataset manifest with:

- file paths/object keys
- file sizes
- last modified timestamps
- checksum/hash if feasible

If the manifest changes, invalidate downstream outputs.

### CDN Cache

Generated visual assets should be served with stable URLs:

```text
/previews/{dataset_id}/{artifact_version}/preview.png
/cloud/{dataset_id}/{artifact_version}/segmentation.png
/reconstruction/{session_id}/{run_id}/optimized.png
```

Stable versioned paths allow long CDN cache without stale content.

## Frontend Performance Improvements

- Use route-level loading states.
- Lazy-load heavy subpage images.
- Keep image containers dimensioned to avoid layout shift.
- Use thumbnails in cards, full images only in viewers.
- Avoid loading comparison assets until the comparison page opens.
- Use `next/image` only for normal web images; for authenticated/signed geospatial outputs, normal `<img>` may be simpler.
- Split large client pages into smaller components after the hackathon for maintainability.
- Avoid unnecessary repeated calls to mission-control and workflow endpoints.

## Backend Performance Improvements

- Add background queue.
- Add object storage abstraction.
- Add PostgreSQL indexes.
- Use raster windows/overviews.
- Avoid repeated full-band reads.
- Reuse generated masks across services.
- Store derived arrays temporarily as compressed rasters if reused.
- Use lower-resolution previews for browser display.
- Keep full-resolution GeoTIFF for export only.
- Add timeouts and memory guards.
- Add structured logs with session_id, dataset_id, and job_id.

## Observability Plan

Minimum production observability:

- Request logs with latency and status code.
- Job logs with stage duration.
- Error tracking with Sentry or equivalent.
- Metrics for queue depth and job runtime.
- Disk/object storage usage metrics.
- Database slow query logs.
- Health checks for API, DB, queue, storage.

Important dashboards:

- API p95 latency
- API error rate
- active jobs
- failed jobs
- average preview generation time
- average cloud pipeline time
- average reconstruction time
- storage usage
- DB CPU/connections

## Load Testing Plan

Before claiming 500-user readiness:

1. Seed 5-10 demo datasets.
2. Precompute common outputs.
3. Use k6, Locust, or Artillery to simulate browsing traffic.
4. Separately simulate heavy job submissions.
5. Measure API p95 latency, queue time, worker memory, DB locks.
6. Tune worker concurrency.
7. Add autoscaling only after measuring real bottlenecks.

Example traffic mix:

- 60% GET dataset/mission-control/workflow pages.
- 20% preview image reads.
- 10% cloud/reconstruction/confidence status reads.
- 5% export/report downloads.
- 5% POST job submissions.

## Fastest Path To Smooth Demo Performance

For the final presentation:

1. Pre-register demo datasets.
2. Pre-run inspection, metadata, preview, cloud, temporal, reconstruction, confidence, and export pipelines.
3. Keep one small dataset available for live processing.
4. Use cached/precomputed outputs for large 5GB+ scenes.
5. Keep backend warm before judging.
6. Keep local Docker fallback ready.
7. Avoid live 5GB uploads during judging.

## Key Rule

Scale the product by scaling the workflow system, not by making every HTTP request do more work. Fast status APIs plus queued processing is the core production design.
